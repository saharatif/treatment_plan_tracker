# Evaluation Plan — 10 Orbs to Better

This document defines how each part of the system is evaluated for correctness before it's
considered done, per the Definition of Done in [`../team/DEVELOPER.md`](../team/DEVELOPER.md)
("basic tests exist for new functionality"). It complements `PROGRESS.md` (what's built) and
`OBSERVABILITY.md` (what's monitored in production).

The sample document `backend/scripts/docs/Marcus_Elliot_Treatment_Plan_v2.pdf` (generated by
`backend/scripts/treatment_plan_generator.py`) is the project's **golden fixture** — it has a known,
hand-authored set of 10 orbs, diagnoses, dates, and billing codes, so every pipeline stage can
be checked against ground truth.

---

## 1. Ingestion Pipeline (Module 2)

### 1.1 OCR (`services/ocr.py`)

| Check | Method |
|-------|--------|
| Text extraction completeness | Run OCR on the `_processing.pdf` variant of the golden fixture; diff extracted text against a hand-transcribed reference for the 10 orb blocks, diagnoses, and header fields (`plan_id`, `patient token`, `plan_start`, `next_visit`). |
| Free vs. paid parity | Run both Tesseract and Mistral OCR on the same fixture; confirm both produce text from which the parser can reconstruct all 10 orbs (exact transcription need not match). |
| Failure handling | Feed a corrupted/empty PDF; confirm `extract_pdf_text` raises a handled error rather than returning silently-empty text that would pass validation. |

### 1.2 LLM Parser (`services/parser.py`)

| Check | Method |
|-------|--------|
| Schema conformance | Output of `extract_orbs_from_text()` must match the JSON shape `validate_parsed_plan` expects: `plan_id`, `patient_id`, `plan_start`, `provider`, `next_visit`, `diagnoses`, `orbs` (10 items). Validate with a Pydantic model in CI against the golden fixture's OCR output. |
| Orb count & numbering | Exactly 10 orbs, `orb_number` 1–10 with no gaps/duplicates — this is also enforced by `validate_parsed_plan`, but should be asserted directly against the golden fixture's known 10 orbs. |
| Catalog matching accuracy | For each of the 10 golden orbs, `match_catalog_orb()` should resolve to the expected `app.orbs.catalog_code` (e.g. golden Orb 1 → `LAB-01`). Track a confusion table per category (Lab, Medicine, Vitamins, Exercise, Monitoring, Diet, Referral, Review) — misclassification across categories is worse than within-category. |
| Few-shot regression | Since `ORB_PARSER_SYSTEM_PROMPT` includes the orb catalog as few-shot examples (per `../agents/02-ingestion-and-billing.md` Step 3), any prompt edit should be re-run against the golden fixture to confirm catalog matching doesn't regress. |
| Free vs. paid parity | Run the same OCR text through GPT-4o and Ollama/Llama 3.1 8B; compare orb extraction accuracy. Free stack is allowed to be weaker but must still pass `validate_parsed_plan` on the golden fixture. |

### 1.3 Validation Gate (`services/validation.py`)

| Check | Method |
|-------|--------|
| Valid plan passes | Golden fixture's parsed output passes `validate_parsed_plan()` with `(True, [])`. |
| Missing-field detection | Mutate the parsed plan (drop `plan_id`, `patient_id`, `plan_start` individually) — confirm each produces a specific error message and `(False, [...])`. |
| Orb-count detection | Remove/duplicate an orb — confirm "Expected 10 orbs, found N" or "Orb numbers not 1-10" fires. |
| Review-queue routing | Confirm a failed validation never reaches `store_plan()` — it must be routed to the review queue (integration test on `POST /api/ingest`). |

## 2. Billing Quotation (Module 2, Step 5)

| Check | Method |
|-------|--------|
| Code pricing | For the golden fixture's known billing codes, `build_quotation()` returns `line_items` with `unit_price`/`medicare_rate` matching the seeded `billing.billing_codes` values. |
| Unmatched codes flagged | Inject a code not present in `billing.billing_codes` — confirm it appears in `unmatched`, not silently dropped, and the quote still completes. |
| Diagnosis vs. billable split | Confirm ICD-10 diagnosis codes land in `diagnoses`, not `line_items` (per `is_billable` flag). |
| Prior-auth surfacing | For codes with `requires_auth = true` (e.g. knee replacement, MRI, colonoscopy/endoscopy, CPAP, skin substitute), confirm they appear in `needs_auth`. |
| Total correctness | `total` equals the sum of `line_items[*].unit_price` — test with a fixture containing a mix of billable/non-billable/unmatched codes. |
| Token-only email | Render the quote email template and assert it contains the token (`PAT-...`) but never a name, DOB, or other PII field. |

## 3. Checkpoint State Machine (Module 3)

The state machine in [DESIGN.md §6](DESIGN.md#6-checkpoint-state-machine) has a small,
enumerable set of transitions — test each one as a table-driven unit test against
`evaluate_plan_checkpoints()`:

| Day | Orbs complete | Expected state | Test name |
|-----|---------------|-----------------|-----------|
| 11 (checkpoint 1) | < 8 | stays `ACTIVE`, warning alert logged | `test_checkpoint1_warns_below_threshold` |
| 11 (checkpoint 1) | ≥ 8 | stays `ACTIVE`, no alert | `test_checkpoint1_no_alert_above_threshold` |
| 14 (checkpoint 2) | 10/10 | `COMPLETED` | `test_checkpoint2_completed` |
| 14 (checkpoint 2) | 8–9 | `IN_GRACE`, hard_stop +7 days | `test_checkpoint2_in_grace` |
| 14 (checkpoint 2) | < 8 | `EXTENDED`, hard_stop +7 days, outreach alert | `test_checkpoint2_extended` |
| 18 (checkpoint 3) | < 10 | unchanged, final warning + escalation alert | `test_checkpoint3_escalation` |
| 21 (checkpoint 4, hard stop) | any | `CLOSED`, orbs locked, report generated | `test_checkpoint4_closes_plan` |

Additional scenarios:

- **Idempotency** — running `evaluate_plan_checkpoints()` twice on the same day for the same
  plan must not duplicate alerts or double-apply state transitions.
- **Locked orbs** — after `CLOSED`, `complete_orb()` on that plan must reject the update.
- **Grace-week completion scope** — once Open Design Question #2 (DESIGN.md §6) is resolved,
  add a test asserting which orbs are eligible for completion during `IN_GRACE`.

## 4. API & Frontend (Module 4)

| Check | Method |
|-------|--------|
| Auth enforcement | Every `/api/*` route from [DESIGN.md §8](DESIGN.md#8-api-surface) returns 401/403 without a valid JWT, and enforces role (clinician vs. billing) where applicable. |
| Dashboard accuracy | `GET /api/dashboard` completion counts match `app.patient_orbs` state for a seeded set of plans across all five `PlanStatus` values. |
| At-risk query | `GET /api/at-risk` returns exactly the plans with <5 orbs complete and ≤7 days remaining — test boundary values (5 orbs/7 days should *not* match). |
| Closure report | `close_plan()` on the golden fixture produces a report PDF listing all 10 orbs with final status, matching the plan's `app.patient_orbs` state at hard stop. |

## 5. End-to-End / MVP Acceptance

Per the MVP cut line ([DESIGN.md §9](DESIGN.md#9-build-plan)), the end-to-end smoke test is:

1. Upload the golden fixture's `_processing.pdf` to `POST /api/ingest`.
2. Confirm a `pending_enrollment` plan is stored with a quotation in `billing.quotation_log`.
3. Call `POST /api/plans/{plan_id}/confirm-billing` → plan becomes `ACTIVE` with 10
   `app.patient_orbs` rows, each linked via `catalog_orb_id`.
4. Mark orbs complete via `POST /api/orbs/{orb_ref}/complete`; confirm dashboard reflects new
   counts.
5. Manually invoke the daily checkpoint job for a plan whose dates have been backdated past
   each checkpoint; confirm state transitions match the table in §3.
6. (Once Module 4 closure ships) Reach hard stop and confirm `CLOSED` + report generation.

This flow should be automated as an integration test once Modules 1–3 + frontend (the MVP cut
line) are implemented, and re-run before any release.

---

## 6. Evaluation Cadence

- **Per-PR** — unit tests for the touched module (table-driven tests above) run in CI.
- **Per-module completion** — run the golden-fixture flow for that module manually, record
  results in `PROGRESS.md`.
- **Pre-MVP-demo** — full end-to-end flow (§5) against the golden fixture.
- **Prompt changes** — any edit to `ORB_PARSER_SYSTEM_PROMPT` re-runs §1.2 catalog-matching
  accuracy against the golden fixture before merge.
